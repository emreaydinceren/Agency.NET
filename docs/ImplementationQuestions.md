# Implementation Questions (Memory Workstream)

Open clarification questions raised by sub-agents during the implementation of `Memory-ProjectPlan.md`. Each entry should be answered by the human reviewer before the affected task can proceed.

**Status (2026-06-02): all IQ-1–IQ-4 accepted/closed.** The chosen defensible defaults are ratified by the owner; no items remain pending.

| ID | Raised by (Task) | Date | Question | Resolution |
|----|------------------|------|----------|------------|
| IQ-1 | SA-C / C.0 | 2026-05-28 | **Reference Direction — circular reference**: The plan (Option B) recommends adding a project reference from `Agency.Memory.Common` → `Agency.Memory.Retrieval` so `MemoryHookFactory.Build()` can directly use `RetrievalEngine` and `RetrievalGate`. However, `Agency.Memory.Retrieval` already holds a project reference to `Agency.Memory.Common` (it uses `IMemoryStore`, `MemoryOptions`, `RankingFormula`, etc.). Adding the reverse reference creates a circular dependency that .NET build rejects. **Chosen resolution (defensible default):** `MemoryHookFactory.Build()` accepts the pre-built retrieval callback as a `Func<Context, CancellationToken, Task>` parameter. `Agency.Memory.Distiller` (which references both `Common` and `Retrieval`) supplies this callback from `RetrievalEngine.RetrieveAsync` gated by `RetrievalGate.ShouldRetrieveAsync`. The `AddAgencyMemory()` DI extension lives in `Agency.Memory.Distiller/DependencyInjection/` rather than in `Agency.Memory.Common/DependencyInjection/`. This is documented in `MemoryHookFactory.Build()` via a `<remarks>` XML doc. | **Accepted (owner, 2026-06-02).** Delegate-injection approach ratified — it ships, builds clean, and avoids the circular reference. `AddAgencyMemory()` stays in `Agency.Memory.Distiller`. No further action. |
| IQ-2 | SA-E / E.3 | 2026-05-28 | **Sub-agent invocation — `RunAsync` is internal**: `Agent.RunAsync` is `internal` in `Agency.Harness` and therefore not accessible from `Agency.Memory.Consolidator`. The spec (§6.3) says the Consolidator is itself an Agent using the existing harness. **Chosen resolution (defensible default):** Use `ChatSession.SendAsync(prompt, ct)` which is `public` and internally delegates to `Agent.ChatAsync → Agent.RunAsync`. This is functionally equivalent for a single-turn consolidated pass (the prompt is the first user message, the agent runs to completion). If SA-G E2E tests expose any behavioural gap (e.g. session-start event semantics), `Agency.Harness` can add `[InternalsVisibleTo("Agency.Memory.Consolidator")]`. | Proceeding with `ChatSession.SendAsync`. |
| IQ-3 | SA-G / G.0 | 2026-05-28 | **Internal access for functional tests**: `IConversationManagerRegistry` and `ILlmClientAdapter` are `internal` to `Agency.Memory.Distiller`. The functional test project needs to (a) register sessions in the registry so `DistillerBackgroundService` can find turns, and (b) implement stub adapters for G.4 crash-recovery tests. **Chosen resolution:** Added `[assembly: InternalsVisibleTo("Agency.Memory.Functional.Test")]` to `Agency.Memory.Distiller/AssemblyInfo.cs`. Consistent with existing test project convention. | Proceeding with `InternalsVisibleTo`. |
| IQ-4 | SA-G / G.1 | 2026-05-28 | **Session coupling between ChatSession and Distiller**: `ChatSession` owns the `InMemoryConversationManager` privately. The Distiller reads from `IConversationManagerRegistry`. There is no hook that auto-registers the session's manager. For G.1, we test the pipeline directly: build a conversation, register it in the registry, enqueue a `DistillationJob`, wait for `DistillationCompletedEvent`, then use `IMemoryStore.SearchAsync` to verify the Fact was written. We then invoke the `RetrievalEngine` directly with a second `Context` to assert recall. This exercises all critical components (LLM call, embedding, Postgres write, retrieval gate) without needing the full ChatSession integration. | Proceeding with direct pipeline testing for G.1. |
